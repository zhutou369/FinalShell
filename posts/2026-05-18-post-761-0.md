## FinalShell 安全最佳实践

安装仅仅是开始，持续的安全实践是保障您服务器和数据安全的关键。

### 1. 强密码与 SSH 密钥认证

*   **不再使用弱密码：** 为 FinalShell 内部设置和所有服务器连接使用复杂且唯一的强密码。
*   **优先使用 SSH 密钥：** 强烈建议为您的所有 SSH 连接配置 SSH 密钥认证。这比密码更安全，因为私钥不会通过网络传输。FinalShell 支持生成和管理 SSH 密钥。
    *   在 FinalShell 中生成密钥对。
    *   将公钥上传到您的服务器 (`~/.ssh/authorized_keys` 文件)。
    *   配置 FinalShell 使用私钥连接。

### 2. 保持 FinalShell 更新

定期检查 FinalShell 的官方网站或通过其内置的更新机制，确保您始终运行最新版本。更新通常包含：

### 4. 理解 CPU 使用率差异

许多用户可能会注意到 FinalShell 显示的 CPU 使用率与服务器面板（如宝塔面板、cPanel 等）显示的数据存在差异。这通常不是 FinalShell 的问题，而是数据采集和计算方式不同造成的。

![Comparing FinalShell CPU Usage with Server Panel Data](/static/images/photo-1498050108023-c5249f4df085.jpg)

*   **采集机制：** FinalShell 通常通过 SSH 连接获取 `/proc/stat` 或其他系统文件的数据，并进行实时计算。服务器面板可能使用不同的 API、守护进程或采集频率。
*   **平均时间：** FinalShell 默认可能显示更实时的瞬时数据，而面板可能显示较长时间的平均值。
*   **虚拟化开销：** 在虚拟化环境中，主机操作系统和虚拟机之间的数据可能存在差异。
*   **闲置与活跃：** 有些工具可能将空闲进程的CPU占用也计算在内，而另一些则只关注活跃进程。

遇到这种情况，建议：
*   **交叉验证：** 使用 `top`, `htop`, `vmstat` 等命令在服务器端直接查看，与 FinalShell 的数据进行比对。
*   **理解工具：** 了解每个工具的数据采集和报告机制。
*   **关注趋势：** 重要的不是数字完全一致，而是它们变化的趋势是否一致，这有助于您判断服务器负载情况。

### 5. SSH 服务器端安全配置

即使 FinalShell 本身再安全，如果您的 SSH 服务器端配置不当，也可能存在风险。

*   **禁用 root 用户 SSH 登录：** 通过 `PermitRootLogin no` 配置。
*   **更改默认 SSH 端口：** 将默认的 22 端口更改为其他非标准端口，增加扫描难度。
*   **限制登录尝试次数：** 通过 `MaxAuthTries` 限制密码尝试次数。
*   **使用防火墙：** 配置服务器防火墙 (如 `ufw`, `firewalld`)，只允许特定 IP 或 IP 段连接 SSH 端口。
*   **禁用密码登录：** 如果您已经全面部署了 SSH 密钥认证，可以考虑禁用密码登录 (`PasswordAuthentication no`)，进一步提升安全性。
