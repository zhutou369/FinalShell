## 为什么 FinalShell 会在高延迟环境下卡顿？

在深入优化之前，我们首先需要理解卡顿的根源。高延迟不仅仅是网络速度慢，更重要的是数据包在客户端和服务器之间往返所需的时间（Round-Trip Time, RTT）长。这会影响到 SSH 会话的方方面面：

*   **TCP 握手与连接建立：** 每次建立新的连接（无论是 SSH 主会话还是 SFTP 子连接），都需要经过三次 TCP 握手。高 RTT 意味着建立连接的时间更长。
*   **SSH 协议开销：** SSH 协议本身为了安全，需要进行加密、解密、密钥交换等操作，这些计算过程虽然耗时很短，但在高延迟环境下，每次交互都会放大这个微小的时间开销。
*   **交互式操作：** 你输入的每一个字符、服务器返回的每一个字符，都需要经历一个 RTT。如果 RTT 达到数百毫秒，那么每次按键都会有明显的延迟感。
*   **文件传输：** SFTP 协议基于 SSH，文件传输的效率受限于 TCP 窗口大小、拥塞控制算法以及网络带宽。高延迟会使得 TCP 窗口无法有效利用，导致传输速度远低于理论带宽。
*   **服务器负载：** 如果海外服务器本身的 CPU 或内存资源紧张，也会加剧响应延迟。
*   **本地网络环境：** 即使海外服务器和骨干网表现良好，你本地的网络状况（例如 Wi-Fi 信号不佳、内网拥堵）也可能成为瓶颈。

了解这些背景后，我们就可以有针对性地进行优化了。

## FinalShell 客户端本地优化

优化，往往从我们最能掌控的客户端开始。

### 调整 SSH 会话参数

FinalShell 提供了丰富的会话配置选项，合理利用它们能显著改善体验。

1.  **启用 TCP KeepAlive (心跳包)**
    高延迟环境下，SSH 会话可能因为长时间不活跃而被中间路由或防火墙断开。开启 `KeepAlive` 可以让客户端定期发送心跳包，保持连接。
    在 FinalShell 的连接配置中，找到“SSH”或“高级”选项卡，勾选“`启用 TCP KeepAlive`”并设置一个合适的间隔（例如 30 或 60 秒）。
    `ClientAliveInterval` 这个参数是针对服务器端的，我们将在后面服务器端优化部分提及。

2.  **合理使用数据压缩**
    SSH 协议支持数据压缩 (`Compression`)。对于带宽较低、延迟较高的网络，开启压缩能有效减少传输的数据量，从而提升感知速度。但如果你的带宽非常充裕，且服务器 CPU 性能一般，压缩反而可能因为 CPU 额外开销而降低速度。
    建议在高延迟、低带宽（例如海外 VPS 常见 100Mbps 带宽但实际使用中速度较慢）的环境下尝试开启此选项。在 FinalShell 连接配置中找到相应选项并勾选。

3.  **选择更轻量级的加密算法 (Ciphers & MACs)**
    SSH 支持多种加密和消息认证算法。一些更现代的算法（如 `chacha20-poly1305@openssh.com`）在性能上优于传统算法（如 `aes256-cbc`）。FinalShell 允许你在连接配置中自定义加密算法。你可以尝试调整顺序或选择更高效的算法。

    *   **推荐的加密算法 (Ciphers)：** `chacha20-poly1305@openssh.com`, `aes256-gcm@openssh.com`, `aes128-gcm@openssh.com`
    *   **推荐的消息认证码 (MACs)：** `hmac-sha2-512-etm@openssh.com`, `hmac-sha2-256-etm@openssh.com`

    请注意，这需要在客户端和服务器端都支持才能生效。

### 优化客户端显示与操作

FinalShell 是一款功能强大的终端，但一些美观的特性在高延迟下可能成为负担。

1.  **关闭不必要的图形功能和动画**
    FinalShell 默认可能开启一些终端美化效果。在高延迟下，每次终端刷新都可能带来自我感知的卡顿。检查 FinalShell 的“设置”菜单，禁用一些不必要的动画、光标闪烁效果或透明度设置。

2.  **调整终端滚动缓冲区**
    如果你的终端输出量巨大，滚动缓冲区过小或过大都可能影响性能。适当调整终端缓冲大小，使其既能满足查看历史命令的需求，又不会占用过多资源。

3.  **尽量使用快捷键而非鼠标**
    在高延迟环境下，鼠标操作（如拖选文本、点击右键菜单）可能会比键盘输入响应更慢。熟练使用 FinalShell 的快捷键，能显著提升操作效率。

### 本地网络环境检查与优化

你的本地网络是连接海外服务器的第一道关卡。

1.  **优先使用有线连接**
    Wi-Fi 相比有线连接，更容易受到信号干扰、丢包和不稳定性影响。如果条件允许，请务必使用网线连接路由器。

2.  **检查本地网络质量**
    使用 `ping` 命令测试到常见国内大型网站的延迟，或者使用网络测速工具检查上传下载速度和抖动。如果本地网络本身就不稳定，那么连接海外服务器会更加困难。

3.  **考虑使用代理/加速服务**
    如果直连海外服务器延迟过高，可以尝试使用高质量的 SOCKS5 代理、SSH 隧道或专业的网络加速服务。FinalShell 本身支持通过代理连接 SSH 服务器，这可以有效规避某些运营商的国际出口瓶颈。你可以在 [FinalShell 首次服务器连接](cn-finalshell-first-server-connection) 的教程中找到相关代理设置的指导。

## 服务器端 SSHD 配置优化

服务器端的 SSH Daemon (sshd) 配置也对连接性能有重要影响。通常，这些配置位于 `/etc/ssh/sshd_config` 文件。修改后需要重启 SSH 服务 (`sudo systemctl restart sshd` 或 `sudo service sshd restart`)。

### 启用或调整 TCP Keepalive

与客户端类似，服务器端也需要配置 Keepalive，以防止服务器主动断开不活跃的连接。
在 `sshd_config` 中添加或修改：

```
ClientAliveInterval 30
ClientAliveCountMax 5
```

这意味着如果客户端在 `ClientAliveInterval` 秒内（这里是30秒）没有发送任何数据或响应，sshd 会向客户端发送一个消息请求响应。如果连续 `ClientAliveCountMax` 次（这里是5次）请求都没有得到响应，sshd 就会断开连接。这有助于保持连接活跃，避免 [FinalShell 连接超时](cn-finalshell-connect-timeout-fix) 问题。

### 禁用 DNS 反向解析

默认情况下，sshd 可能会对传入的连接执行 DNS 反向解析，将客户端 IP 地址解析为域名。这在高延迟环境下会显著增加连接建立的时间。
在 `sshd_config` 中设置：

```
UseDNS no
```

这将禁用反向解析，加速连接过程。

### 调整认证方式

密码认证需要多次交互验证，而 SSH 密钥对认证则只需一次密钥交换，效率更高且更安全。强烈建议使用 [FinalShell SSH 密钥对登录](cn-finalshell-ssh-keypair-login)。不仅能提升连接速度，还能大大增强安全性。

### 服务器资源管理

确保你的海外服务器本身有足够的资源。
*   使用 `top` 或 `htop` 命令检查 CPU 和内存使用率。
*   检查磁盘 I/O 是否正常 (`iostat`)。
*   清理不必要的服务，释放资源。

## 提升文件传输效率 (SFTP)

文件传输通常是高延迟下最令人头疼的问题。

### FinalShell 的 SFTP 优化

FinalShell 内置的 SFTP 客户端通常表现不错，但仍可优化：

*   **多线程传输：** 对于传输大量小文件，多线程传输能显著加速。FinalShell 默认会利用多线程，但你可以检查相关设置是否启用。
*   **调整传输缓冲区大小：** 有些客户端允许调整 SFTP 传输的缓冲区大小，理论上更大的缓冲区可以减少交互次数，提升吞吐量。FinalShell 内部通常已进行优化，但在特殊情况下可以尝试寻找高级设置。
*   **拖拽上传注意事项：** [FinalShell 的 SFTP 拖拽上传功能](cn-finalshell-sftp-drag-upload) 方便快捷，但在传输大文件时，务必注意网络稳定性，避免中途断开导致文件损坏或需要重新上传。

### 服务器端文件同步工具

对于需要频繁同步大量文件或大文件的情况，使用 FinalShell 的 SFTP 界面可能不是最有效率的方法。可以考虑以下命令行工具：

*   **`rsync`：** `rsync` 是一款非常强大的文件同步工具，它可以在增量传输、压缩传输、断点续传等方面表现出色，尤其适合在高延迟网络环境下进行大量文件同步。你可以在 SSH 终端中直接使用 `rsync` 命令。
*   **`screen` / `tmux`：** 如果传输时间很长，担心 SSH 会话断开导致传输中断，可以在服务器端使用 `screen` 或 `tmux` 创建一个会话，然后在其中执行文件传输命令。这样即使 SSH 会话断开，传输任务仍在服务器后台继续运行。

## 高级技巧与辅助工具

除了上述常规优化，还有一些高级技巧和工具可以进一步改善体验。

### TCP BBR 加速

Google 开发的 TCP BBR 拥塞控制算法能有效提升 TCP 连接在长距离、高延迟、高带宽链路上的吞吐量。
在 Linux 内核 4.9 及以上版本中，可以轻松开启 BBR。
在服务器上执行以下命令（以 Ubuntu/Debian 为例）：

1.  编辑 `/etc/sysctl.conf`，添加以下两行：
    ```
    net.core.default_qdisc=fq
    net.ipv4.tcp_congestion_control=bbr
    ```
2.  执行 `sudo sysctl -p` 使配置生效。
3.  验证是否生效：`sysctl net.ipv4.tcp_congestion_control`，如果返回 `net.ipv4.tcp_congestion_control = bbr` 则表示成功。

BBR 对于改善文件传输速度和整体会话响应速度都有显著帮助。

### Mosh (Mobile Shell)

Mosh 是一款专为移动和高延迟网络环境设计的 SSH 替代品。它使用 UDP 协议，具有以下优点：

*   **IP 漫游：** 即使 IP 地址改变（例如从 Wi-Fi 切换到蜂窝网络），会话也能保持活跃。
*   **瞬时响应：** Mosh 会预测你输入的内容，并在本地立即显示，大大减少了高延迟带来的感知卡顿。
*   **弹性：** 即使网络波动导致丢包，Mosh 也能很好地恢复会话。

虽然 FinalShell 目前暂不支持直接集成 Mosh 客户端，但作为一种补充方案，你可以考虑在服务器上安装 Mosh 服务端，并在本地使用 Mosh 客户端（如 Mosh on Linux/macOS/Windows Subsystem for Linux）进行连接。

## 常见问题

### Q1: FinalShell 总是连接超时怎么办？

A1: 连接超时是一个常见问题，通常与网络可达性、防火墙或 SSH 服务配置有关。首先，请检查你的本地网络是否正常，服务器 IP 地址和端口（默认为 22）是否正确。其次，确保服务器端的防火墙（如 `ufw` 或 `firewalld`）已开放 SSH 端口。最后，检查服务器的 SSH 服务是否正常运行，以及 `sshd_config` 中是否配置了 `ListenAddress` 或其他限制。你可以参考我们之前发布的 [FinalShell 连接超时解决](cn-finalshell-connect-timeout-fix) 的详细教程，一步步排查解决。

### Q2: 为什么我优化后感觉变化不大？

A2: 优化技巧可以缓解高延迟带来的影响，但无法彻底消除物理距离导致的固有高 RTT。如果你尝试了所有优化方法后仍然感觉变化不大，可能核心问题是：
*   **物理距离过远：** 例如，连接北美服务器和连接亚洲服务器的延迟天生就不同。
*   **网络运营商限制：** 你的本地运营商或国际出口运营商对流量进行了 QoS 限制或路由不佳。
*   **服务器地理位置选择不当：** 尝试选择离你物理位置更近的海外服务器。
在这些情况下，考虑使用高质量的代理或专业的国际网络加速服务，可能比单纯调整 SSH 参数更有效。

### Q3: FinalShell 版本更新对优化有帮助吗？

A3: 当然有！FinalShell 开发团队会不断优化软件性能，修复已知问题，并可能引入新的功能或改进现有功能。新版本通常包含更优化的网络传输机制、更稳定的连接管理和更高效的资源利用。因此，我强烈建议您及时更新 FinalShell 到最新版本，以获得最佳的使用体验。您可以访问 [FinalShell 官方下载](cn-finalshell-official-download-channel) 渠道，确保获取到最新、最稳定的版本。
